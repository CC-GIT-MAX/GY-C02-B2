# 信号总线使用指南

## 一句话规则

> **跨模块可变状态 → 信号；模块私有常量 → 头文件 `static const`；一次性事件 → 日志 + flag，不进信号表。**

## 判定流程

```
需要被另一个模块读取？
  ├─ 否 → 模块内部 static 变量
  └─ 是 → 值会随时间变化？
              ├─ 否 → 模块内 static const，或共享头里的常量
              └─ 是 → 写入是周期性？（采样/计算结果）
                       ├─ 是 → 信号
                       └─ 否 → 一次性事件 → 用日志或模块私有 flag
```

## 反模式

| 反模式 | 后果 | 正确做法 |
|---|---|---|
| `extern uint8 power_state;` 出现在多个 .c | 谁写谁读搞不清，单元测试做不了 | `Signal_Set/Get(SIG_IGN_ON, ...)` |
| 写一个 `Signal_Set(SIG_SPEED, ...)` 然后立刻在同一函数读 | 没用，局部变量即可 | 用局部变量 |
| 任何模块都向 `SIG_TT_LEFT_TURN` 写 | 谁覆盖谁说不清 | 单一所有者；其他模块通过事件去"请求"，所有者决策 |
| 把 `SIG_MAX` 当数组下标 | 越界 | 用 `Signal_Get(id)`，内部已经边界检查 |

## 单一所有者原则

每个信号应该有**唯一的所有者**（写入方），其他模块只能 `Get`。

例：
- `SIG_VEH_SPEED_KPH_X10` 所有者：`mod_can_rx`（从 CAN 解析）
- `SIG_FUEL_LEVEL_PCT` 所有者：`mod_fuel`（从 ADC 读电阻换算）
- `SIG_IGN_ON` 所有者：预留（KL15 IO 中断，待 `mod_power` 落地）

## 何时新增信号

1. 至少 2 个模块需要它
2. 不适合作为函数参数传递
3. 没有"近实时"要求（信号读取本身是 1-cycle 内存读，不需要保护）

## 何时**不要**新增信号

1. 只在 1 个模块内使用 → 用 `static` 变量
2. 调用即响应（如"切换菜单") → 走 `Event_Post`（后续批次）或函数调用
3. 数据量大（> 16 字节） → 用共享 buffer + 信号通知，而不是把整个 buffer 塞进 signal
